# 用编程 Agent 安装并配置 StillOn

StillOn 面向已经在使用 Codex 或 Claude Code 的用户。复制下面的 Prompt 并粘贴给你的编程 Agent；它会从官方仓库安装 StillOn，并继续带你完成适合自己场景的访问配置和第一条消息。

<!-- prompt:start -->
请从 StillOn 官方仓库安装并完成首次配置：
https://github.com/bzbj/stillon

你是这次安装和首次使用的向导。不要只把服务启动当作“完成”：先让本机可用，再按我的使用场景完成配置，最后带我在 StillOn 里发出第一条消息。

开始前，请用简短问题确认三件事：
1. 这台实例的 **Machine Name（机器名称）**；
2. 我这次想先用 Codex、Claude Code，还是两者；
3. 安装后我想怎样访问：A. 只在这台电脑上；B. 同一局域网；C. 外网。

如果我暂时没有选择访问方式，先完成 A，再明确问我是否要继续配置 B 或 C。不要替我猜测，也不要默认为我想开放网络。

请按以下步骤执行：
1. 在做任何修改前，阅读仓库中的 README、SECURITY.md 和与当前平台有关的安装文档。
2. 检查这台电脑的操作系统、Shell、Bun 是否可用及其版本，以及是否已有 StillOn 安装。简要说明计划后，再按仓库的官方步骤执行。
3. 保留已有的用户数据和配置。未经询问，不要覆盖、移除或停掉已有安装。
4. 第一次启动 StillOn 前，使用我给出的 Machine Name。在第一次启动前将 `STILLON_MACHINE_NAME` 写入 StillOn 专用配置；若已有设置文件和机器名称，必须保留原值，并告诉我可在 **Settings → General → Machine Name** 中修改。
5. 在本地构建并启动 StillOn，验证 health endpoint 和 `http://127.0.0.1:3210`（或实际端口）上的浏览器应用。报告准确的本地访问地址、安装 revision/version、进程状态、日志位置、更新方式，以及回滚或卸载步骤。
6. 检查 Codex 和 Claude Code 的可用性，但只要求我选择的 provider 通过首次检查：
   - 报告 `codex` 与 `claude` 是否存在及版本；不要擅自改动任一 provider 的配置。
   - 对 Codex，使用 `codex login status` 检查认证状态；若未认证，说明交互式 `codex login` 的下一步，并在启动浏览器登录或设备认证前先征得我的同意。
   - 对 Claude Code，按它的官方方式检查认证状态；若未安装或未认证，说明需要的交互式登录，并在开始前先征得我的同意。
   - 不要因为我暂时不用的 provider 未安装而阻塞 StillOn；准确说明缺少的前置条件即可。不得索取、复制或输出任何凭据、浏览器验证码、会话 token 或 API key。
7. 本机验证通过后，继续带我完成我选择的访问方式：
   - **A. 只在本机：** 保持 `127.0.0.1` 绑定，给出准确 URL，等我确认已经在浏览器打开。
   - **B. 同一局域网：** 先解释这会让同一网络内的设备可以访问本机项目、终端和 Agent。得到明确同意后，使用仓库文档对应的 `--remote` 和强密码方式启动，找出这台电脑的局域网 IPv4，给出准确的 `http://局域网-IP:端口` 地址。先在本机验证监听状态，再请我用另一台设备实际打开确认。未经确认，不要修改防火墙、路由器或 Tailscale；若它们阻塞访问，说明原因和下一步。
   - **C. 外网：** 只在我明确选择后继续。先说明这相当于把本机的项目、终端和 Agent 交给远程浏览器控制，必须同时做好 StillOn 密码和 Cloudflare Access。引导我注册或登录自己的 Cloudflare 账号、准备已接入 Cloudflare 的域名，并在 Cloudflare Dashboard 创建受管理的 Tunnel 和公开 hostname，使其指向本机的 `http://localhost:3210`。再创建 Cloudflare Access 的 self-hosted application 和只允许我的策略。不要用 `--remote`、直接暴露本机端口，或自动使用临时 `--share` Tunnel；除非我明确要求临时公开地址。
   - 对 C，只有在我已在本机安全地输入 Tunnel token 后，才按仓库文档使用 `--cloudflared <token>` 和 StillOn 密码启动命名 Tunnel。不要要求我把密码、Tunnel token、Cloudflare 登录信息或验证码贴到聊天中，也不要把它们写进 Shell 历史、日志、源代码或配置文件。如果当前工具无法在不泄露密钥的情况下接收它，停在这一步并告诉我如何在本机安全输入。
   - 外网配置完成后，用公开 hostname 和 Cloudflare Access 从独立浏览器或设备实际验证；不要只凭本机进程启动就宣布成功。
8. 访问方式验证完成后，带我完成第一条消息：
   - 根据我选择的 provider，在 StillOn 里新建对话。发送前先展示建议的消息并征得我的确认，因为它可能消耗额度或计费。
   - 如果我还没有具体任务，建议使用这条无副作用的开场消息：`请确认你已连接 StillOn。在不修改任何文件、不执行命令的前提下，用一句话说明你可以怎样帮助我。`
   - 如果你能操作浏览器，就在 StillOn 中发送并确认收到回复；如果不能，就告诉我精确的点击路径，等我确认结果。没有收到回复时，说明具体阻塞点和安全的下一步，不要声称首次配置已完成。
9. 如果本次安装是替换 Kanna，在 StillOn 的本机健康检查和我选择的 provider 首次检查通过前，保留 Kanna 及其数据不动。仅在检查通过后，再单独说明切换或关停选项并征得确认；Kanna 绝不能成为 StillOn 的运行时依赖。
10. 只有在说明影响并征得确认后，才提供可选的“按用户启动的后台服务”。

安全约束：
- 默认保持 StillOn 仅绑定 `127.0.0.1`；网络暴露必须由我选择并逐步确认。
- 在执行任何需要高权限、具有破坏性、会修改防火墙/DNS/已有服务，或会在 Cloudflare 创建资源的操作前，先说明影响并请求确认。
- 外网访问不要只依赖一个入口：命名 Cloudflare Tunnel 要配 Cloudflare Access，StillOn 也要使用强密码；不要直接暴露 origin port。
- 不要在 Shell 历史、日志、源代码或聊天输出中泄露密钥。
- 把 provider 测试和第一条消息视为可能消耗额度或计费的操作；只有在我明确同意后才能发送。
<!-- prompt:end -->

## 这个 Prompt 对“完成”的定义

编程 Agent 是安装和首次使用的向导，不是一次性安装器。一次成功的运行必须有五个可观察结果：命名完成的 StillOn 实例、健康的本地服务、我选择的 provider 已就绪、我选择的访问方式已实际验证，以及一条经我同意后发出的第一条消息。

即使用户只需要本机访问，也要在本机验证后明确说明局域网和外网是可选的下一步；只有用户选择后才继续配置。
